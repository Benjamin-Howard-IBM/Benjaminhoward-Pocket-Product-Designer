MCSP Vault offering needs to provide access to audit logs of customer
vault clusters. This RFC documents where these logs are stored and how
customers can access them.

## Background

Vault is being offered on [*FedRAMP certified MSCP
platform*](https://www.fedramp.gov/marketplace/products/F1208031461A/)
on AWS GovCloud and there is a requirement to provide the capability to
store audit logs of customer vault clusters and give customers access to
them in a FedRAMP compliant way. The requirement is called out in [*this
PRD*](https://ibm.sharepoint.com/:w:/r/sites/hermes/_layouts/15/Doc.aspx?sourcedoc=%7B00635D24-6CB8-4960-BA33-0DFD2091362B%7D&file=HCPV-Vault%20on%20MCSP%20-%20FedRAMP%20Logging%20Approach.docx&action=default&mobileredirect=true&DefaultItemOpen=1).

Currently, [*customer vault clusters are run in Openshift
cluster*](https://ibm.sharepoint.com/:w:/r/sites/hermes/_layouts/15/Doc.aspx?sourcedoc=%7BA03B5201-906E-49D6-8A77-7C28A23A1422%7D&file=VLT-Vault%20on%20MCSP.docx&action=default&mobileredirect=true)
in a separate namespace (per cluster) and the audit logs are stored in
persistent volume that is backed by EBS. Vector is configured as side in
the vault pods and it can be configured to stream the logs out to
customer's cloudwatch instance in their AWS account. This implementation
was done to mirror HVD setup and is documented
[*here*](https://ibm.sharepoint.com/:w:/r/sites/hermes/_layouts/15/Doc.aspx?sourcedoc=%7BA93EEEEC-AD65-4E73-83D5-CEE42A2CC701%7D&file=HCPV-011_-_MSCP_Vault_Streaming.docx&action=default&mobileredirect=true).
While this setup is ok in commercial offering it is not allowed in
FedRAMP due to strict network boundary. The recommendation from FedRAMP
COE (center of excellence team) is to let customers pull the logs. This
is how currently other products provide audit logging capability.

MSCP has
[*documentation*](https://pages.github.ibm.com/ibm-saas-platform/MultiCloud-SaaS-Framework/Onboarding_Dev/phase2_1_fed_tooling/logging/)
about adopting audit logging but it was later learnt that is primarily
for internal audit logs that are sent to QRadar for long term analysis.
They have setup for [*external audit
logs*](https://github.ibm.com/ibm-saas-platform/external-logging) but
that is not yet supported in FedRAMP environment.

## Proposal

### Storage

Currently we plan to store two pieces of customer data for each cluster:
snapshots and audit logs. Both of this information will be stored in
separate S3 buckets. There will be two buckets per cluster created in
Vault AWS account -- one for snapshots and one for audit logs:

- instance-{instance-id}-snapshots
- instance-{instance-id}-audit-logs

These buckets along with any policies associated with them will be
created by cluster operator as it is provisioning the cluster. The
bucket will be tagged with subscription id to allow fetching all logs
belonging to a customer. The snapshot operator will be updated to store
snapshots in these buckets instead of *mcsp-vault-snapshot-{env}* common
bucket currently.

The snapshot bucket will be deleted when the cluster is getting deleted
whereas the audit logs bucket will remain due to requirement to keep
them for 1 year or so (confirm from Sara). Cleanup of audit log buckets
will require separate automation.

This was chosen due to following reasons:

- Audit logs buckets will be accessed by customers whereas snapshots are
  accessed by service only. Having separate buckets can avoid accidental
  leakage to customers.
- Audit logs buckets have different storage requirements like object
  lock and retention period which can be set easily at bucket level. It
  is possible to set policies with prefixes but the chances of error are
  higher.
- In future each bucket can be encrypted with customer provided or
  separate key if needed. This is not possible with shared buckets.
- The isolation is easy to explain to auditors.

### Limits

AWS's default limit for number of buckets is 10000 but can be extended
to 1 million on request. Even with 10000 there can be 5000 clusters
provisioned at the same time which we are not going to reach anytime
soon.

### Abandoned Ideas 

There is spectrum of tenant isolation strategy starting from least
isolation to most:

- One bucket for all data
- One bucket for snapshots and another for audit logs
- One bucket for snapshots and separate buckets for audit logs
- One bucket per subscription containing both audit logs and snapshots
- One bucket per instance containing both audit logs and snapshots
- One bucket per instance for audit logs and another bucket per instance
  for snapshots

It was decided to use the maximum isolation strategy for reasons
mentioned above and due to the nature of targeting high compliance with
less number of tenants.

## Customer access

Customers will be provided direct access to audit logs S3 bucket via
cross account role chaining. The process will be as follows for a
cluster:

On audit logging screen when the access is disabled, customer will be
shown following information:

- Access disabled
- AWS role ARN in our account (named *instance-{id}-audit-reader*) that
  role in their AWS account will have to assume
- External ID used when assuming the role
- Instructions on how to create role in their AWS account

The AWS role in our account will be configured to be assumed by any role
and will not have any permissions. Its trust policy will look like this:

+----------------------------------------------------------------------+
| > {                                                                  |
| >                                                                    |
| > \"Version\": \"2012-10-17\",                                       |
| >                                                                    |
| > \"Statement\": \[                                                  |
| >                                                                    |
| > {                                                                  |
| >                                                                    |
| > \"Sid\": \"CustomerAssumeRole\",                                   |
| >                                                                    |
| > \"Effect\": \"Allow\",                                             |
| >                                                                    |
| > \"Principal\": {                                                   |
| >                                                                    |
| > \"AWS\": \"\*\"                                                    |
| >                                                                    |
| > },                                                                 |
| >                                                                    |
| > *\"Action\": \"sts:AssumeRole\",*                                  |
|                                                                      |
| *\"Condition\": {*                                                   |
|                                                                      |
| *\"StringEquals\": {*                                                |
|                                                                      |
| *\"sts:ExternalId\": \"fa6050b4-6e45-4cad-89da-4ac621251f1a\"*       |

To enable access, customer will be asked to provide a role ARN whose
trust policy will allow assuming to role *instance-{id}-audit-reader* in
our AWS account. The ARN will point to a role in their AWS account. This
is how their role's trust policy will look like:

+-------------------------------------------------------------------------+
| > {                                                                     |
| >                                                                       |
| > \"Version\": \"2012-10-17\",                                          |
| >                                                                       |
| > \"Statement\": \[                                                     |
| >                                                                       |
| > {                                                                     |
| >                                                                       |
| > \"Sid\": \"AssumeVendorLogReadRole\",                                 |
| >                                                                       |
| > \"Effect\": \"Allow\",                                                |
| >                                                                       |
| > \"Principal\": {                                                      |
| >                                                                       |
| > \"AWS\":                                                              |
| > \"arn:aws:iam::\<VAULT_ACCOUNT_ID\>:role/instance-{id}-audit-reader\" |
| >                                                                       |
| > },                                                                    |
| >                                                                       |
| > \"Action\": \"sts:AssumeRole\"                                        |
| >                                                                       |
| > }                                                                     |
| >                                                                       |
| > \]                                                                    |
| >                                                                       |
| > }                                                                     |
+-------------------------------------------------------------------------+

The customer provided role will then be added to our role's trust policy
(allowing customer provided role to assume our role) and the role's
permission will be updated to provide read access to instance's bucket
objects. The screen will now show the following information:

- Access is enabled
- Our role ARN *instance-{id}-audit-reader* that customer's role has to
  assume
- Customer's role ARN
- External ID
- Instructions / sample commands to assume roles and get logs from s3
  bucket
- TBD / nice-to-have: Possibly last time the customer's role was used

### Management APIs

To support the above workflow the following APIs will be implemented in
management service.

#### GetAuditLoggingAccess

Response when access is enabled

+----------------------------------------------------------------------+
| {                                                                    |
|                                                                      |
| "enabled": true,                                                     |
|                                                                      |
| "log_reader_role": "instance-{id}-audit-reader role ARN",            |
|                                                                      |
| "customer_role_arn": "\<customer provided role ARN\>",               |
|                                                                      |
| "assume_role_external_id": "\<random string used by customer to      |
| assume role\>"                                                       |
|                                                                      |
| }                                                                    |
+----------------------------------------------------------------------+

Response when the access is disabled

+----------------------------------------------------------------------+
| {                                                                    |
|                                                                      |
| *"enabled": false*                                                   |
|                                                                      |
| *"log_reader_role": "instance-{id}-audit-reader role ARN",*          |
|                                                                      |
| "assume_role_external_id": "\<random string used by customer to      |
| assume role\>"                                                       |
|                                                                      |
| }                                                                    |
+----------------------------------------------------------------------+

#### EnableAuditLoggingAccess

Request:

+----------------------------------------------------------+
| {                                                        |
|                                                          |
| "customer_role_arn": "\<customer provided role ARN\>",   |
|                                                          |
| }                                                        |
+----------------------------------------------------------+

Response:

+----------------------------------------------------------------------+
| {                                                                    |
|                                                                      |
| "enabled": true,                                                     |
|                                                                      |
| "log_reader_role": "\<system role ARN\>",                            |
|                                                                      |
| "customer_role_arn": "\<customer provided role ARN\>",               |
|                                                                      |
| "assume_role_external_id": "\<random string used by customer to      |
| assume role\>"                                                       |
|                                                                      |
| }                                                                    |
+----------------------------------------------------------------------+

#### DisableAuditLoggingAccess

Empty request and response

### MCSP Role

The IAM action associated with above management APIs will be
*audit-streams.manage* and the following roles will have this action in
it which means customers with these roles can access audit logs:

- Auditor
- Operator
- Security Engineer
- Administrator
