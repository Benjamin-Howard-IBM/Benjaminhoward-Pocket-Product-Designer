# Vault Next Cryptography Dashboard - Wireframes

Depends on: ../02.strategy/Vault-Next-Combined.md, 005.26.05.19.PDR-WireframePlan-VaultNextDashboards.md

## Persona served

Primary: System Security Engineer. Owns key lifecycle posture, algorithm strategy, rotation cadence, and PQC readiness.

Secondary: Platform Engineer. Configures key stores and routes rotation operations.

## Day 0 goals

- Inventory view of keys and key stores at the selected scope.
- Surface state (active / decrypt-only / disabled / pending deletion) and type (symmetric / asymmetric / MAC).
- No time-series, no PQC migration UI, no algorithm trend.

## Day 0 interactions

- Project and Key Store dropdowns scope all regions.
- Type multi-select filters the breakdown panels and the implied keys count; the Key Stores table is not filtered by type (rows are stores).
- Row click on Key Stores: navigates to the Key Store detail page (out of scope here).
- DataTable toolbar search filters store rows by name.
- Empty state when no key stores in scope: `EmptyState` inside the table body with title "No key stores in this project" and a single ghost `Button` "Create a key store".

## Day 1 goals

- Add temporal context (age histogram, rotations due).
- Surface algorithm distribution and versions-per-key distribution.
- Introduce a gated PQC readiness callout that explains what will land later without committing to UI patterns.

## Day 1 interactions

- DatePicker range gated on Prometheus per Filtering Requirements CSV. If unavailable: disable picker, show `InlineNotification`, hide sparklines, replace age histogram and versions-per-key histogram with `EmptyState`.
- Row click on Key Stores table: opens side flyout with store detail (same pattern as Secrets dashboard; see 007). Out of scope to detail here.
- Row OverflowMenu: "Open key store", "Schedule deletion", "Export inventory". "Schedule deletion" requires a Carbon `Modal` confirmation; destructive.
- Rotations due "View list ->": routes to a filtered Keys list (out of scope to wire).
- PQC readiness callout has no interaction at Day 1. It is a "Coming soon" notification with body copy. No buttons. The intent is to set expectations, not to invite clicks.
- Algorithm panel empty state: when no keys in scope, `EmptyState` "No keys configured in this scope".

## Carbon mapping

Components used:

- `UI Shell`, `Breadcrumb`, page title.
- `Dropdown` (Org, Project, Key Store), `FilterableMultiSelect` (Type), `DatePicker` range (Day 1 only), `Tag`.
- `Tile` (non-clickable on this dashboard).
- `DataTable` with `Toolbar` (search, filter at Day 1), `Pagination`, `OverflowMenu` for row actions (Day 1).
- `SimpleBarChart`, `DonutChart`, `HistogramChart` (Day 1).
- `InlineNotification` (kind=info) for the Prometheus gating copy and the PQC readiness "Coming soon" copy.
- `EmptyState` for zero-data panels.
- `Button` ghost for "View list ->"; `Button` primary not used at this fidelity.
- Side panel pattern (Carbon for Products) for the optional key-store flyout at Day 1.

Token slots:

- Surface `$background`; panel `$layer-01`; subtle border `$border-subtle-01`.
- PQC readiness uses `$layer-02` to visually separate from the main inventory band without color.
- Numeric value `$text-primary`; labels and captions `$text-secondary`; helper copy `$text-helper`.
- Spacing: tile padding `$spacing-05`; section gutter `$spacing-07`.

Grid columns (Carbon 16-column):

- Tile row: 4 / 4 / 4 / 4.
- PQC readiness panel: 16, sitting directly under the tile row to anchor the page's posture story.
- DataTable: 16.
- Day 0 breakdowns (type / state): 8 / 8.
- Day 1 paired panels (algorithm / age, versions / rotations, type / state): 8 / 8 each.

## Open questions / assumptions

- Open: Should PQC readiness sit above or below the Key Stores table? Day 1 ASCII puts it above to make the differentiator visible without clicking. If stakeholders prefer to keep the dashboard inventory-first, move it below the breakdowns.
- Open: "# Key Versions" tile aggregation. Sum across all keys is straightforward, but at scale this can be a large number. Confirm whether a median-versions-per-key metric is more useful and whether the tile should display both.
- Assumption: Key store status taxonomy (available / pending deletion) mirrors secret store taxonomy. Strategy doc does not state this explicitly. Confirm.
- Assumption: "Rotations due" derives from key age plus an org rotation policy stored somewhere (Manage scopes settings, label assumption). If no policy source exists, this panel must collapse to a gated `EmptyState`.
- Assumption: Algorithm list (AES-256, RSA-2048, ECDSA-P256, Ed25519, HMAC-SHA256, ML-KEM-768) is illustrative. Real algorithm set is system-defined; confirm with the cryptography service team.
- Open: ML-KEM-768 appears in the Day 1 algorithm panel but PQC readiness is gated as "Coming soon". Resolve the inconsistency: either show ML-KEM rows in the algorithm panel as evidence of PQC-capable keys, or strip them out until the PQC readiness panel ships. Recommend keeping them; the algorithm panel is descriptive, the readiness panel is prescriptive.
