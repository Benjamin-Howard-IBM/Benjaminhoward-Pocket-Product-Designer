# Vault Next Secrets Dashboard - Wireframes

Depends on: ../02.strategy/Vault-Next-Combined.md, 005.26.05.19.PDR-WireframePlan-VaultNextDashboards.md

## Persona served

Primary: Platform Engineer. Owns secret store configuration, project-level inventory, and remediation of secrets approaching TTL.

Secondary: Application Developer. Read-only view of which stores and secret types are available in their projects (consumer summary).

## Day 0 goals

- Inventory view of secrets and secret stores at the selected scope.
- Project-scoped filters. Single-store drill is link-only at Day 0 (table row links to a store detail page, not an inline flyout).
- No time-series; no operational metrics.

## Day 0 metric mapping

| Region | HashiConf metrics row |
|---|---|
| Tile: # Secrets | Secrets Metrics > # of Secrets |
| Tile: # Secret Stores | Secrets Metrics > # Stores (available / pending deletion) |
| Tile: # Clients (consuming) | Secrets Metrics > # Clients by type (human / machine), combined count at Day 0 |
| Tile: # Pending deletion | Secrets Metrics > # of Secrets by status (pending deletion) + # Stores (pending deletion) |
| DataTable: Secret Stores | Secrets Metrics > # of Secrets by store. Per-row count = secrets in that store. |
| Panel: Secrets by type | Secrets Metrics > # of Secrets by type |
| Panel: Secrets by status | Secrets Metrics > # of Secrets by status |

Out of scope and not shown at Day 0: CREATED BY column on stores or secrets (Secrets CSV marks out of scope), LAST ACCESSED column (out of scope), all operational traffic metrics.

## Day 0 interactions

- Project dropdown: scopes all tiles, the Stores table, and breakdown panels.
- Store dropdown: when a specific store is selected, the Stores table collapses to one row and the breakdown panels recompute for that store only.
- Type multi-select: filters the breakdown panels and the implied secrets count; the table is unchanged because rows are stores, not secrets.
- Row click on Stores table: navigates to the Secret Store detail page (out of scope here). Day 0 has no inline drill.
- DataTable toolbar search: filters store rows by name.
- Empty state when no stores in scope: `EmptyState` inside the table body with title "No secret stores in this project" and a single ghost `Button` "Create a secret store".

## Day 1 goals

- Add temporal context (age histogram, TTL approaching list, recently rotated).
- In-table drill from a store row into a side-panel flyout (no full page navigation).
- Surface the link between auth methods and secret access.

## Day 1 metric mapping

| Region | HashiConf metrics row |
|---|---|
| All Day 0 tiles + sparklines | unchanged + time-series derivation (gated) |
| DataTable Age column | Secrets Metrics > age of secrets (TTL, time since creation), aggregated per store (median) |
| Panel: Secrets by age | Secrets Metrics > age of secrets (TTL, time since creation), histogram |
| Panel: TTL approaching | Secrets Metrics > age of secrets (TTL). Derived: TTL - now < 7d and < 30d. |
| Panel: Recently rotated | Labeled assumption. No direct CSV row. Derives from Secret Version state transitions (primary -> inactive) and creation timestamps in the strategy doc's "Version Lifecycle Strategy" section. |
| Panel: Auth methods used to access secrets | Users/Entities/Identities > # Auth methods, # accounts per method, with By Auth Method filter (Filtering Requirements CSV). Gated on access-log instrumentation; treat as labeled assumption until access logging is confirmed in scope. |
| Flyout: top 10 secrets by age | Secrets Metrics > # of Secrets by store + age of secrets |

Still out of scope at Day 1: CREATED BY and LAST ACCESSED columns on the per-secret tables (Secrets CSV out-of-scope note). They do not appear, not even as empty placeholders.

## Day 1 interactions

- DatePicker range gated on Prometheus per Filtering Requirements CSV. If unavailable: disable the picker, show `InlineNotification`, hide sparklines, replace age histogram with `EmptyState` ("Age histogram requires time-series data"), and reduce TTL approaching panel to a single message.
- Row click on Stores table: opens Carbon side panel (flyout) with store detail. Closing returns focus to the originating row.
- OverflowMenu per row: "Open store", "Schedule deletion", "Export inventory (CSV)". "Schedule deletion" is destructive; show Carbon `Modal` confirmation. Out of scope to wire here.
- TTL approaching "View list ->": routes to a filtered Secrets list with TTL filter applied (out of scope to wire).
- Recently rotated empty state: when zero rotations in the selected range, show `EmptyState` "No rotations in this range".
- Auth methods panel empty state: when access logging not configured, `InlineNotification` "Linking auth methods to secret access requires access logging. Configure under Manage scopes > Observability."

## Carbon mapping

Components used:

- `UI Shell`, `Breadcrumb`, page title.
- `Dropdown` (Org, Project, Store), `FilterableMultiSelect` (Type), `DatePicker` range (Day 1 only), `Tag`.
- `Tile` (non-clickable on this dashboard; tiles already reflect the current scope and do not deep-link further).
- `DataTable` with `Toolbar` (search, filter at Day 1), `Pagination`, `OverflowMenu` for row actions (Day 1).
- `SimpleBarChart`, `HistogramChart` (Day 1).
- Side panel pattern for the store flyout (Carbon for Products side panel; falls back to a stacked layout if not available).
- `InlineNotification` (kind=info) for gating copy.
- `EmptyState` for zero-data panels.
- `Button` ghost for "View all ->", "View list ->", "Open store ->"; `Button` primary not used at this fidelity.

Token slots:

- Surface `$background`; panel `$layer-01`; flyout overlay `$layer-02`; subtle border `$border-subtle-01`.
- Numeric value `$text-primary`; labels and captions `$text-secondary`; helper copy `$text-helper`.
- Spacing: tile padding `$spacing-05`; section gutter `$spacing-07`; flyout inset `$spacing-05`.

Grid columns (Carbon 16-column):

- Tile row: 4 / 4 / 4 / 4.
- DataTable: 16.
- Day 0 breakdowns (type / status): 8 / 8.
- Day 1 paired panels (age / TTL, recently rotated / auth methods, type / status): 8 / 8 each.
- Side flyout: 6-column overlay anchored to the right.

## Open questions / assumptions

- Open: At Day 1, should "# Clients (consuming)" split into human / machine on this dashboard or remain combined? Central separates them. Recommend keeping combined here to reduce tile row width; reconcile with Open Question 1 in 005.
- Open: Age aggregation per store. Median, p50, p95? Wireframe uses "median". Confirm with engineering and the System Security Engineer persona.
- Assumption: "Recently rotated" maps to Secret Version state transitions per the strategy's "Version Lifecycle Strategy" section. Confirm there is a queryable rotation event source.
- Assumption: Auth methods panel sources from access-log linkage. If access logs are not in scope for the dashboard MVPs, this panel collapses to a gated empty state at Day 1.
- Open: TTL approaching windows. 7d and 30d are placeholders. Confirm with Platform Engineer in usability review.
