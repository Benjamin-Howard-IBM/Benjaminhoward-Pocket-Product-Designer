# Vault Next Observability Dashboards - Wireframe Plan

Depends on: ../02.strategy/Vault-Next-Combined.md (sections: "Vault Next Side Bar Nav", "HashiConf Dashboard Metrics (Reference)", "New structure", "Personas, JTBDs, CUJs, Sitemap")

## Goals

- Define the wireframe scope for three Vault Next observability dashboards: Central, Secrets, Cryptography.
- Lock Day 0 (MVP) vs Day 1 (expanded) scope so downstream wireframes do not drift.
- Map every tile and panel back to a specific row in the HashiConf Dashboard Metrics reference, or label it as an explicit assumption.
- Establish shared shell components (UI Shell, scope picker, filter bar, tile grid) so the three dashboards stay consistent.

## Non-goals

- Visual fidelity. All wireframes are gray, ASCII, low fidelity.
- High-fidelity Carbon Charts styling. Day 0 uses static bar/donut; Day 1 sketches line/histogram placement only.
- Access Control management surfaces (Identities, Groups, Roles, Grants, Auth methods, Accounts CRUD). Counts only on the Central dashboard.
- Audit log surfaces under Support.
- Agent-specific telemetry, PQC migration interaction flow, and operational traffic instrumentation (deferred per CSV note "ALL OPERATIONAL METRICS OUT OF HASHI SCOPE").
- Secret CREATED BY and LAST ACCESSED columns (called out as out of scope in the secrets metrics CSV).
- Mobile breakpoints. Carbon 16-column grid at desktop only for this pass.

Application Developer is not a primary persona for any observability dashboard. Their JTBD ("quickly see all credentials available to me") is served by per-resource list views, not aggregate dashboards. Day N+1 may add a developer "my access" dashboard. Out of scope here.

## Day 0 vs Day 1 scope rules

Day 0 (MVP):

- Static or near-static metric tiles. No time-series.
- Scope picker (Organization / Project). One scope selected at a time.
- Basic filters: By Project, By Store (per Filtering Requirements CSV).
- Counts only. No trend.
- Simple lists and tables (DataTable with search + pagination, no batch actions).
- Static breakdowns rendered as Carbon `SimpleBarChart` or `DonutChart` with fixed snapshots.
- No operational traffic, no agent telemetry, no PQC migration UI.
- Out-of-scope CSV items (CREATED BY, LAST ACCESSED, all operational metrics) are absent. They do not appear as empty states either.

Day 1 (expanded):

- Adds time-range filter via Carbon `DatePicker` (range). Gated on Prometheus or equivalent time-series store. If unavailable, show `InlineNotification` (info) explaining the dependency and disable the picker.
- Trend sparklines on tiles where the underlying metric is countable over time.
- Drill-downs from Secret Store / Key Store table rows into a detail flyout (Carbon side panel pattern).
- Status breakdowns (active / decrypt-only / disabled / pending deletion for keys; status field for secrets).
- Age histograms for secrets and keys (Carbon `HistogramChart`).
- Top stores by activity (gated on operational metrics; empty-state copy if not wired).
- "What needs attention" surface: keys pending deletion, secrets nearing TTL, rotations due.
- Period-over-period comparison strip on Central.
- PQC readiness callout on Cryptography Day 1, gated as "Coming soon" placeholder with copy explaining what it will surface.

## Shared components

Defined once here; referenced by 006 / 007 / 008.

### App shell

- Carbon `UI Shell`: `Header`, `SideNav`, `HeaderGlobalAction`.
- `SideNav` items follow the "Vault Next Side Bar Nav" section: Secrets (Secrets, Secret Stores), Cryptography (Keys, Key Stores), Access control (Identities, Groups, Roles, Grants, Authentication methods, Accounts), Manage scopes, Support (Audit logs).
- Page header: Carbon `Breadcrumb` (Organization > [Project]) + page title.

### Scope picker

- Two `Dropdown` controls in the page header, immediately under the breadcrumb:
  - `Dropdown: Organization`
  - `Dropdown: Project` (includes an "All projects" sentinel)
- Selection updates every tile, panel, and table on the page. No partial scope loading.
- Active scope echoes back as Carbon `Tag` chips below the dropdowns when not at the default.

### Filter bar

- Day 0 filters: `Dropdown: Project`, `Dropdown: Store`. Optional `FilterableMultiSelect: Type` on Secrets and Cryptography.
- Day 1 adds: `DatePicker (range): Time range`. Gated control.
- Active filters render as `Tag` chips with dismiss icons, sitting between the filter bar and the content grid.

### Tile row

- Carbon `Tile` (clickable variant when the tile deep-links).
- Day 0: large numeric value, label, small caption (scope context like "in Project: payments-prod").
- Day 1: adds inline sparkline (top-right of tile) and a delta value vs previous period.
- Grid: 4-up tile row = each tile spans 4 columns of the 16-column grid.

### Time-range picker (Day 1 only)

- `DatePicker` with range mode. Presets: 1h, 24h, 7d, 30d, custom.
- Gated by Prometheus (or equivalent time-series store) availability. If not configured, render disabled and show `InlineNotification` (kind=info, low-contrast) with body: "Time-range filtering requires a time-series datastore (Prometheus). Configure under Manage scopes > Observability to enable."

### Empty / gated states

- Carbon `InlineNotification` for dependency-gated panels (Prometheus, operational metrics, PQC).
- Carbon for Products `EmptyState` pattern inside panels with no data: short title, one-line body, optional single ghost `Button` for the configuration path.

### Tokens (grayscale fidelity)

- Surface: `$background`, panel `$layer-01`, panel hover `$layer-hover-01`.
- Text: `$text-primary` for values, `$text-secondary` for captions and labels, `$text-helper` for empty-state copy.
- Spacing: tile padding `$spacing-05`, section gutters `$spacing-07`, grid column gutter from the Carbon 16-col grid.
- Borders: `$border-subtle-01` for tile and panel edges in low-fi.

### Grid spans

- Tile row: four `Tile` at 4 columns each = 16 columns.
- Side-by-side summary panels (Central): two panels at 8 columns each.
- Wide table (Stores): 16 columns, full width.
- Side-by-side breakdowns (Secrets by type / by status, Keys by type / by state): two panels at 8 columns each.
- Attention panel (Day 1, Central): 16 columns, single row band above the tile row, or 8 columns alongside the comparison strip.

## Open questions

1. Source of truth for "# Clients by type (human / NHI / agent)". The Users/Entities/Identities CSV uses human/NHI/agent; the Secrets CSV uses human/machine. Resolve taxonomy before the Clients tile row ships.
2. Is "pending deletion" a true status, or a derived state from soft-delete grace period? Affects whether we can filter and count it directly or compute it client-side.
3. "By Auth Method" and "By Entity" filters are listed in the Filtering Requirements CSV. Day 0 only ships By Project and By Store. Confirm Day 1 wave for Auth Method / Entity filters.
4. "Most/least used stores or scopes" depends on operational traffic instrumentation. CSV marks all operational metrics out of Hashi scope. Day 1 wireframes treat this as a gated panel. Confirm whether to ship the panel skeleton at all if Prometheus is not on the roadmap.
5. PQC readiness panel content. We have the Cryptography differentiator (PQC migration, ML-KEM rotation) but no surfaced metric definition. Day 1 placeholder lists candidate metrics (% keys on PQC-ready algorithm, # keys flagged for migration). Needs PM sign-off before fidelity rises.
6. Scope picker behavior when "All projects" is selected. Do per-project tables roll up by project, or flatten? Recommend roll-up with project column visible.
7. Clickable tiles deep-link target. From Central, "Total Secrets" tile should route to the Secrets dashboard pre-filtered to the current scope. Confirm route and state-preservation behavior with engineering.
